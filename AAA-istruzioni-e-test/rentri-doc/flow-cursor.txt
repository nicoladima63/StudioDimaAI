Feedback di comprensione
Il documento descrive l’interoperabilità RENTRI tramite API, che permette di:
Trasmettere dati di registro cronologico di carico/scarico e dei formulari FIR.
Accedere a servizi di vidimazione virtuale, anagrafiche, firma remota, e tabelle di codifica.
Le API sono suddivise in vari servizi:
anagrafiche: accesso ai dati anagrafici delle unità locali e registri.
ca-rentri: gestione firma remota e device associati.
codifiche: accesso alle tabelle di codifica.
dati-registri: trasmissione delle registrazioni annotate nei registri cronologici.
formulari: trasmissione dati FIR per rifiuti pericolosi.
vidimazione-formulari: produzione e vidimazione virtuale dei FIR.
Le API saranno disponibili in produzione dal 15/12/2024 in modalità “STUB” (solo test di collegamento, risposte simulate), e poi progressivamente aperte.
Modalità STUB:
Le chiamate GET a /status restituiscono 422 e risposta vuota.
Altre GET: 200 e risposta vuota.
POST/DELETE/PUT: 422.
Se serve autenticazione e fallisce: 401.
Flusso di lavoro step by step
1. Analisi delle specifiche API
Esaminare i file YAML/OpenAPI presenti nella cartella rentri-doc per ogni servizio elencato (anagrafiche, ca-rentri, codifiche, dati-registri, formulari, vidimazione-formulari).
Annotare endpoint, parametri richiesti, autenticazione, payload e risposte attese.
2. Preparazione ambiente di sviluppo
Configurare variabili ambiente/locali per gestire le credenziali e i parametri di autenticazione richiesti dalle API RENTRI.
Predisporre una struttura di progetto (es. cartella rentri/ o modulo dedicato) per isolare il codice di integrazione.
3. Implementazione chiamate di base (modalità STUB)
Implementare le chiamate GET agli endpoint /status di ciascun servizio per verificare la connessione (attendersi risposta 422).
Implementare almeno una chiamata GET, POST, PUT, DELETE per ogni servizio, gestendo le risposte simulate (200 o 422).
Gestire l’autenticazione secondo le specifiche (token, certificati, ecc.).
4. Gestione errori e logging
Implementare un sistema di logging per tracciare richieste, risposte e codici di stato.
Gestire i casi di errore previsti (422, 401, ecc.) e documentare i comportamenti.
5. Automazione e test
Scrivere script di test automatici per ogni endpoint, verificando che il comportamento sia conforme alla modalità STUB.
Prevedere la possibilità di passare facilmente da ambiente STUB a produzione reale (quando sarà disponibile).
6. Documentazione operativa
Documentare la procedura di configurazione, esecuzione e test delle API RENTRI.
Annotare eventuali differenze tra modalità STUB e modalità reale, per facilitare la futura transizione.
